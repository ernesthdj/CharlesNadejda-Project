### TICKET-01 : Race condition TOCTOU — SELECT FOR UPDATE dans ConsumeStock
**Agent :** #4 Backend
**Priorité :** P0
**Effort :** 2h
**Dépend de :** aucune
**Fichiers à modifier :**
- `DAL/BomProductionDAL.cs` (lignes 238–252 et 338–408)

**Description :**
`VerifierDisponibilite()` est appelée HORS de la transaction à la ligne 238. Ensuite, `ConsumeStock()` opère EN transaction (lignes 338–408). Entre ces deux appels, un autre utilisateur peut vider le stock. Deux corrections simultanées :

1. Dans `BomProductionDAL.Executer()`, ligne 238 : déplacer l'appel `VerifierDisponibilite()` APRÈS le `BeginTransaction()`, en lui passant la connexion et la transaction en cours.
2. Dans `BomStockDAL.GetLotsDispoFIFO()` et `BomStockDAL.GetBomStocksFIFO()` : ajouter `FOR UPDATE` en fin de requête SELECT pour poser un verrou pessimiste sur les lignes lues.

Alternative si `FOR UPDATE` complexifie trop le refactoring : ajouter un commentaire `// MONO-POSTE UNIQUEMENT — race condition si multi-utilisateurs` sur la ligne 237 et déplacer `VerifierDisponibilite` dans la transaction en lui passant `conn` + `tx` comme paramètres surchargés.

**Critère d'acceptation :**
- [ ] `VerifierDisponibilite` est appelée à l'intérieur du bloc `using (var tx = conn.BeginTransaction())`
- [ ] Aucun SELECT de lot n'est fait sur une connexion séparée après l'ouverture de la transaction
- [ ] Build sans erreur

### TICKET-04 : Null-guard + valeur négative dans UnitConvertisseur.Convertir
**Agent :** #4 Backend
**Priorité :** P0
**Effort :** 15min
**Dépend de :** aucune
**Fichiers à modifier :**
- `Forms/UnitConvertisseur.cs` (ligne ~102 — méthode `Convertir`)

**Description :**
Trois cas limites non gardés dans `Convertir(decimal valeur, string uniteSource, string uniteCible)` :

1. `uniteSource == null` → `NullReferenceException` dans `_versBase[null]`
2. `uniteCible == null` → idem
3. `valeur < 0` → retourne une valeur négative sans erreur, causant une pénurie fantôme (manque toujours = 0 → production lancée à tort)

Ajouter en tête de méthode :
```csharp
if (uniteSource == null) throw new ArgumentNullException(nameof(uniteSource));
if (uniteCible == null) throw new ArgumentNullException(nameof(uniteCible));
if (valeur < 0) throw new ArgumentOutOfRangeException(nameof(valeur), "La valeur à convertir ne peut pas être négative.");
```

**Critère d'acceptation :**
- [ ] Les trois guards sont présents en tête de `Convertir()`
- [ ] Un appel avec `uniteSource = null` lève `ArgumentNullException` (pas `NullReferenceException`)
- [ ] Un appel avec `valeur = -1` lève `ArgumentOutOfRangeException`
- [ ] Build sans erreur

**Critère d'acceptation :**
- [ ] Après une production exécutée, les réservations associées aux lots consommés ont `actif = 0` en DB

### TICKET-09 : Garde anti-cycle dans BomCoutDAL — HashSet<int>

**Priorité :** P1

**Dépend de :** aucune
**Fichiers à modifier :**
- `DAL/BomCoutDAL.cs` (ligne ~144, méthode `CalculerCout` et `CalculerLigneFiche`)

**Description :**
La récursion `CalculerCout() → CalculerLigneFiche() → CalculerCout()` n'a pas de garde contre les dépendances circulaires. Une fiche BOM qui se référence indirectement elle-même provoque un `StackOverflowException`.

Ajouter un `HashSet<int> fichesVisitees` en paramètre de la récursion :
```csharp
private static decimal CalculerCout(int idFiche, int quantite, HashSet<int> fichesVisitees)
{
    if (!fichesVisitees.Add(idFiche))
        throw new InvalidOperationException($"Cycle détecté dans la BOM : fiche id={idFiche} déjà visitée.");
    // ... logique existante ...
}
```
Le point d'entrée public `CalculerCout(int idFiche, int quantite)` crée un nouveau `HashSet<int>()` et délègue à la surcharge privée.

**Critère d'acceptation :**
- [ ] La surcharge privée prend `HashSet<int> fichesVisitees` en paramètre
- [ ] Un cycle est détecté et lève `InvalidOperationException` avec message explicite
- [ ] Le point d'entrée public crée le HashSet et délègue
- [ ] Build sans erreur
