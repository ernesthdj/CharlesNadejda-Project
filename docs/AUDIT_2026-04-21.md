## AGENT #1 — PRODUCT OWNER : Audit Fonctionnel

### Vue d'ensemble

L'application ERP ArtisaStock couvre un périmètre fonctionnel réel et bien structuré pour un projet académique C# WinForms. La refonte en SFA (Single-Form Application — architecture où tout s'affiche dans un unique formulaire principal sans ouverture de fenêtres secondaires) est engagée et partiellement réalisée dans `FrmPrincipal`. Les modules BOM (Bill Of Materials — Nomenclature de production multi-niveaux) de simulation et de production sont fonctionnels. Cependant, l'audit révèle une **coexistence de deux paradigmes incompatibles** : l'ancien modèle `ShowDialog()` cohabite avec la nouvelle architecture SFA, créant des incohérences que l'examinateur pourrait identifier.

Le projet dispose de 22 Forms, d'une documentation riche (11 fichiers docs), et d'un document User Stories (`10_PO_USER_STORIES.md`) produit par une session antérieure. Cet audit se base exclusivement sur le code source.

---

### Ce qui fonctionne bien

- **Hub principal cohérent** : `FrmPrincipal` implémente un tableau de bord avec 4 StatCards (Ingrédients, En alerte, Fiches BOM, Productions 7j), une liste de productions récentes et un panneau d'alertes stock. La navigation par rail gauche est opérationnelle.

- **Écran Contexte complet** : `ShowContexteScreen()` affiche les niveaux du contexte actif avec un `SplitContainer` 50/50 fixe présentant les fiches BOM à gauche et le stock de production à droite. Le bouton Dupliquer fiche (US-07) est implémenté via `BomFicheDAL.Duplicate()`.

- **Simulation + Production unifiées** : `FrmBomProductionSimulation` fusionne correctement simulation et exécution. La simulation colorie les lignes vert/rouge selon les pénuries, bloque le bouton "Lancer" si pénuries > 0, calcule le coût estimé via `BomCoutDAL.CalculerCout()`, et l'exécution est lancée en `async/await` sur `Task.Run()`. La protection contre le double-clic est présente.

- **Vue stock global** : `FrmVueStock` utilise une VIEW MySQL (`vue_stock_global`), applique une coloration DLC (rouge/orange/vert), filtre par activité via des chips, et affiche la quantité réservée.

- **Onboarding utilisateur** : `ShowOnboarding()` guide l'artisan en 4 étapes avec un lien direct vers la création de stock et un CTA (Call-To-Action — bouton d'action principal) vers "Créer ma première activité".

- **Rapport du jour** : `GenererRapport()` produit un `PrintDocument` avec en-tête activité, liste des productions du jour avec coûts, alertes stock, et coût total via `PrintPreviewDialog`.

- **Palette de marque cohérente** : la palette chocolat/crème/or est appliquée dans `FrmPrincipal`, `FrmVueStock`, `FrmArtisaStock` et `FrmBomProductionSimulation`.

### Risques produit

**Risque #1 — Incohérence SFA / modal (CRITIQUE)**
La coexistence de `ShowDialog()` dans `FrmPrincipal` (pour BOM CRUD, Activités, confirmations) et de la navigation inline SFA crée une expérience fragmentée. Pour l'examen, l'examinateur peut constater que la règle R-UX-01 du document US Stories produit par ce projet est violée dans le code livré — contradiction interne difficile à justifier.

**Risque #2 — Code mort non nettoyé**
`FrmArtisaStock`, `FrmBomProduction`, `FrmBomSimulation`, `FrmRecettes`/`FrmRecetteEdit` présentes dans le projet sans rôle clair. Des Forms inutilisées peuvent piéger l'examinateur lors d'une démonstration.

**Risque #3 — Catalogue web non connecté**
7 Forms Catalogue/Parfums/Produits existent mais inaccessibles. Si la grille d'examen PDSGBD pénalise les fonctionnalités déclarées mais non démontrables, ce risque est directement noté.

**Risque #4 — Exceptions silencieuses sur le Hub**
Les `catch { }` vides dans `ShowHubScreen()` font que si la DB est inaccessible, le tableau de bord s'affiche vide sans message.

---

### Recommandations PO

1. Migrer les CRUD BOM vers inline dans `FrmPrincipal` — transformer `ShowDialog(FrmBomContexteEdit/FrmBomNiveauEdit/FrmBomFicheEdit)` en chargement dans `_pnlDroit`.
2. Remplacer `new FrmActivites().ShowDialog()` dans le bandeau par `ShowRessourceScreen(RessourceType.Activites)`.
3. Supprimer ou exclure les Forms orphelines : `FrmArtisaStock`, `FrmBomProduction`, `FrmBomSimulation`, `FrmRecettes`, `FrmRecetteEdit`.
4. Connecter le Catalogue web depuis `FrmPrincipal` — section "CATALOGUE" dans le rail gauche.
5. Remplacer les `MessageBox.Show()` de confirmation par des Panels de confirmation inline.
6. Corriger les `catch { }` silencieux dans `ShowHubScreen()`.

## AGENT #2 — ARCHITECTE : Audit Architecture & Design

### Vue d'ensemble architecturale

Application C# WinForms (.NET Framework 4.8.1) organisée en trois couches explicites : **Models** (POCO — Plain Old C# Objects), **DAL** (Data Access Layer — toutes classes statiques avec ADO.NET brut sur MySQL), et **Forms** (deux classes de base génériques). Un schéma MySQL de 15 tables actives (après migration v12) supporte un BOM générique multi-niveaux.

Le pattern général est un **MVP partiel** (Model-View-Presenter) : `FrmListeBase<T>` et `FrmEditBase` centralisent le comportement CRUD commun, mais la logique métier et les appels DAL restent directement dans les Forms concrètes (pas de Presenter séparé).

---

### Points forts architecturaux

1. **Classes de base génériques bien conçues.** `FrmListeBase<T>` fournit 5 membres abstraits et absorbe tout le comportement CRUD commun. `FrmEditBase` centralise le cycle Valider→Sauvegarder avec `ErrorProvider`.

2. **BOM multi-niveaux générique et extensible.** Le modèle Contexte → Niveaux → Fiches → Lignes permet une hiérarchie de production à profondeur variable. La distinction `TypeInput` dans `BomFicheLigne` (ingredient | fiche) est une conception souple.

3. **FIFO implémenté côté DAL.** `BomStockDAL.GetLotsDispoFIFO` et `GetBomStocksFIFO` trient sur `date_achat ASC` / `date_production ASC`. La consommation dans `BomProductionDAL.ConsumeStock` respecte strictement l'ordre FIFO lot par lot.

4. **Transactions atomiques sur les opérations critiques.** `BomFicheDAL.Insert/Update` et `BomProductionDAL.Executer` utilisent `BeginTransaction / Commit / Rollback` avec gestion d'exception correcte.

5. **UnitConvertisseur robuste et centralisé.** Trois familles (masse/volume/pièce), conversion deux-étapes via unité de base, `InvalidOperationException` sur unités incompatibles.

6. **Dénormalisation contrôlée documentée.** `bom_stocks.id_contexte` et `id_activite` sont explicitement dénormalisés (migration v11) pour éviter des jointures en cascade. Décision sourcée et justifiée.

7. **Prévention des injections SQL.** 100% des appels utilisent `cmd.Parameters.AddWithValue()`. Aucune concaténation SQL détectée.

### Schéma de dépendances observé

```
Forms (FrmXxx hérite FrmListeBase<T> / FrmEditBase)
  │
  ├── appelle directement ──► DAL (BomFicheDAL, BomProductionDAL, ...)
  │                                │
  │                                ├── utilise ──► DbHelper (connexions MySQL)
  │                                ├── instancie ──► Models (POCO)
  │                                └── appelle ──► UnitConvertisseur
  │
  └── appelle ──► UnitConvertisseur (utilitaire transverse)

BomProductionDAL ──► BomNiveauDAL, BomFicheDAL, BomStockDAL
  └─ Couplage fort inter-DAL (devrait être dans une couche Service)
```

---

### Dettes techniques prioritaires

1. **(CRITIQUE — concurrence)** Race condition `VerifierDisponibilite` hors transaction — P6
2. **(ÉLEVÉ — intégrité)** `GetIdNiveauPrecedent` connexion séparée dans une transaction — P3
3. **(MOYEN — maintenabilité)** DAL et Models orphelins post-migration v12 — P7
4. **(MOYEN — performance)** Multiples connexions par opération de production — P2
5. **(MOYEN — testabilité)** Absence d'interfaces sur les DAL — P10
6. **(BAS — organisation)** `UnitConvertisseur` mal placé — P4

---

### Recommandations architecturales

**R1** — Isoler l'orchestration de production dans une couche `ProductionService` qui coordonne les DAL sans ouvrir lui-même des connexions.

**R2** — Calculer `idNiveauPrecedent` **avant** `BeginTransaction` dans `BomProductionDAL.Executer` et le passer en paramètre à `ConsumeStock`.

**R3** — Supprimer les artefacts orphelins post-migration v12 : `Models/Recette.cs`, `Models/Parfum.cs`, `Models/Categorie.cs` et leurs DAL.

**R4** — Documenter la contrainte mono-poste : ajouter `// ATTENTION : mono-poste uniquement — race condition si multi-utilisateurs` sur `BomProductionDAL.Executer:237`.

**R5** — `Forms/UnitConvertisseur.cs` → `Utils/UnitConvertisseur.cs`, namespace `CharlesNadejda.Utils`.

### Score Architecture : 7 / 10

Fondations solides (classes de base, BOM, FIFO, transactions, sécurité SQL), pénalisé par la race condition, les connexions multiples dans `Executer`, les artefacts orphelins et l'absence de testabilité.

## AGENT #3 — UI/UX DESIGNER : Audit Ergonomie & Expérience Utilisateur

**Niveau global : intermédiaire-positif.** Le design system est cohérent (palette chocolat, Segoe UI, FlatStyle uniforme), la structure navigationnelle est logique, et plusieurs bonnes pratiques sont appliquées (confirmation avant suppression, bouton dangereux non par défaut, feedback asynchrone). |---|---|---|---|---|
| Aucun | **Injection SQL** | — | — | **Zéro injection détectée** — 100% des requêtes utilisent `@parametre` via `AddWithValue` |

### Gestion des connexions & ressources

**Point fort majeur** : 100% des connexions utilisent `using (var conn = ...) { using (var cmd = ...) }`. Aucun oubli de `Dispose()` détecté dans les 13 fichiers.

**`BomCoutDAL`** : calcul récursif ouvre une nouvelle connexion à chaque niveau. Pour une hiérarchie 3 niveaux × 5 lignes : 10-15 connexions parallèles lors du calcul. Fonctionnel avec le pool, non optimal.

### Transactions & atomicité

| Méthode | Transaction | Rollback | Tables | Évaluation |
|---|---|---|---|---|
| `BomContexteDAL.InsertAvecNiveaux()` | Oui | `catch { tx.Rollback(); throw; }` | `bom_contextes` + `bom_niveaux` | Correct |

| `BomReservationDAL.Insert()` | Non | — | `bom_reservations` (mono-table) | Acceptable |

**Problème P1 — TOCTOU** (`BomProductionDAL.Executer():238-244`) : `VerifierDisponibilite` hors transaction → `ConsumeStock` dans transaction. Entre les deux, un autre processus peut vider le stock.

**Problème P2 — Reads FIFO sans `FOR UPDATE`** (`BomProductionDAL.ConsumeStock():351-406`) : `GetLotsDispoFIFO()` et `GetBomStocksFIFO()` sont des SELECT sans verrou sur une connexion séparée. En isolation `READ COMMITTED`, un autre thread peut modifier les lots entre la lecture FIFO et le `UPDATE`.

---

### Qualité des algorithmes BOM

- **FIFO correct** : `GetLotsDispoFIFO` trie `date_achat ASC`, déduction des réservations via sous-requête corrélée
- **Stock net correct** : `SUM(lots.quantite_disponible) - SUM(reservations actives)`
- **`GetIdNiveauPrecedent()`** : résolution dynamique du niveau N-1 par requête SQL — propre et générique

**Problème P3 — Réservations non libérées lors de l'exécution** : dans `Executer()`, les lots réservés consommés ne sont pas libérés (`actif = 0` non émis). Les réservations restent actives → réduction artificielle du stock disponible pour les prochaines simulations.

**Problème P4 — Récursion sans garde anti-cycle** dans `BomCoutDAL` : `CalculerCout() → CalculerLigneFiche() → CalculerCout()` sans `HashSet<int>` de fiches visitées. Dépendance circulaire en DB → `StackOverflowException`.

---

### UnitConvertisseur — Analyse

- `StringComparer.OrdinalIgnoreCase` — robuste aux variations de casse ✅
- `Convertir()` lève `InvalidOperationException` sur unités incompatibles — fail-fast correct ✅
- **`_versBase[uniteSource]` sans guard** : si une unité est dans `_groupes` mais absente de `_versBase`, `KeyNotFoundException` non descriptive
- **`UniteBase()` retourne `"piece"` par défaut** pour toute unité inconnue — masque silencieusement les fautes de frappe
- **Null non gardé** : `uniteSource = null` → `NullReferenceException`

---

| `BomProductionDAL.VerifierDisponibilite():107` | 2N requêtes pour N lignes fiche | Moyenne |

---

### Points forts backend

- Zéro injection SQL — 100% `AddWithValue`
- `using` systématiques sur toutes les connexions et commandes
- Pattern Bind/Map cohérent dans tous les DAL
